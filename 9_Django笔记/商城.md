```
接口设计思路:
		1.添加序列化器字段
		2.校验
			2.1 从前端获取图片验证码id和图片验证码内容
			2.2 通过前端图片验证码id从redis数据库中获取图片验证码内容
			2.3 为防止一个图片验证发送多个短信验证码 删除redis中的图片验证码内容
			2.4 比较图片验证码
			2.5判断是否在60s内
```

```python
class ImageCodeCheckSerializer(serializers.Serializer):
    """
    图片验证码校验序列化器
    """
    image_code_id = serializers.UUIDField()
    text = serializers.CharField(max_length=4, min_length=4)

    def validate(self, attrs):
       
        image_code_id = attrs['image_code_id']
        text = attrs['text']
        # 查询真实图片验证码
        redis_conn = get_redis_connection('verify_codes')
        real_image_code_text = redis_conn.get('img_%s' % image_code_id)
        if not real_image_code_text:
            raise serializers.ValidationError('图片验证码无效')

        # 删除图片验证码
        redis_conn.delete('img_%s' % image_code_id)

        # 比较图片验证码
        real_image_code_text = real_image_code_text.decode()
        if real_image_code_text.lower() != text.lower():
            raise serializers.ValidationError('图片验证码错误')

        # 判断是否在60s内
        mobile = self.context['view'].kwargs['mobile']
        send_flag = redis_conn.get("send_flag_%s" % mobile)
        if send_flag:
            raise serializers.ValidationError('请求次数过于频繁')

        return attrs
```

**2.6.2发送短信异步任务实现**

celery 的原理

```python
#为什么使用celery
举个例子:当我们在进行注册时往往需要经过注册信息入库/渲染激活邮件的内容/调用发送邮件的接口/返还响应给浏览器/用户才能继续浏览网页
但是用了celery之后我们可以将渲染激活邮件内容/调用发送邮件接口这样的事情放在任务队列中,从而给用户带来更好的用户体验
#celery定义:
异步分布式任务调度模块
#celery的组成：
客户端发布任务,任务队列存储任务(常见的broker有Redis,rabbitMQ),任务处理者(woker)
#celery的工作原理
Client A发送任务请求给broker；
broker收到请求并存放在任务队列；
如果celery没有空闲的worker则等待，如果这时celery有空闲的worker，则获取任务并执行；
Celery的 worker执行完任务后将执行结果写到Backend results；
Client A通过定时轮询查询执行结果；
#celery怎么用
1.创建一个celery用的包
2.创建一个config.py文件用来指明celery中任务队列broker所使用的redis数据的IP地址
3.创建一个main.py的启动文件
	3.1 为celery使用Django配置文件进行设置
    3.2 创建一个celery的应用
    3.3 导入config.py的配置文件
    3.4 把要执行的异步任务注册到celery中
4.创建一个相应异步任务的包并在包中创建一个tasks.py的文件用来存放所要存放的异步任务
	4.1 在tasks.py这个文件中,需要响应的异步任务添加celery应用的装饰器
5.在要使用异步任务的视图中导入相应异步任务的包就可以用
```

```python
                    'min_length': '仅允许8-20个字符的密码',
                    'max_length': '仅允许8-20个字符的密码',
                }
            }
        }

    def validate_mobile(self, value):
        """验证手机号"""
        if not re.match(r'^1[3-9]\d{9}$', value):
            raise serializers.ValidationError('手机号格式错误')
        return value

    def validate_allow(self, value):
        """检验用户是否同意协议"""
        if value != 'true':
            raise serializers.ValidationError('请同意用户协议')
        return value

    def validate(self, data):
        # 判断两次密码
        if data['password'] != data['password2']:
            raise serializers.ValidationError('两次密码不一致')

        # 判断短信验证码
        redis_conn = get_redis_connection('verify_codes')
        mobile = data['mobile']
        real_sms_code = redis_conn.get('sms_%s' % mobile)
        if real_sms_code is None:
            raise serializers.ValidationError('无效的短信验证码')
        if data['sms_code'] != real_sms_code.decode():
            raise serializers.ValidationError('短信验证码错误')

        return data

    def create(self, validated_data):
        """重写保存方法，增加密码加密"""

        # 移除数据库模型类中不存在的属性
```

###2.9 JWT机制说明

```python
"""
1.JWT:json web token(令牌):记录用户的登录状态

2.传统session的弊端:
				1.每一个用户都需要在服务端做一次记录,而随着认证用户的增多，服务端的开销会明显增大
				2.用户的每次请求都必须在之前认证的服务器上进行限制了负载均衡器的能力限制了应用的扩展能力
				3. 因为是基于cookie来进行用户识别的, cookie如果被截获，用户就会很容易受到跨站请求伪造的攻击。
3.JWT的构成
	3.1 header(头部)+payload(载荷)+signature(签证)
	3.2 header:typ(认证机制的类型)+alg(加密的算法) ----然后将头部base64进行加密
	playload: 标准中注册的声明+公共的声明+私有的声明----(base64后的)
	signature:header (base64后的)
			  payload (base64后的)
			  secret:
	 HMACSHA256(header.payload, 'secret')
4.jwt的流程
		1.注册时前端提交相应数据 后端创建jwt字段
		2.服务器返回jwt给浏览器
		3.浏览器将jwt放置在认证头中传给服务器
		4.服务器校验jwt签证 获取用户信息从jwt中
		5.发送用户信息给客户端
5.jwt的优点
		1.jwt是跨语言的
		2.因为payload的存在 所以可以存储一些其他的所必要的非敏感信息
		3.jwt占用字节小便于传输
		4.不要在服务端保持信息易于扩展
		
		
hmac:Hash-based Message Authentication Code 哈希消息认证码
"""
```

**3. 幂等型与非幂等性**

1.幂等性(传递增量):对于相同的请求，不管请求⼏次，最终的结果都相同，则称为幂等

```
PUT /cart/
sku_id 1
count 2 # 表示结果设置2个
第⼀次访问： 结果 10+2=12
第⼆次访问： 结果 12+2=14
第⼀次访问： 结果 2
第⼆次访问： 结果 2
结果与请求次数⽆关，结果紧跟设置值有关
```

1.⾮幂等(传递结果)

```
PUT /cart/
sku_id 1
count 2 # 表示增加2个

第⼀次访问： 结果 10+2=12
第⼆次访问： 结果 12+2=14
结果与请求次数有关，结果值收到请求的次数影响
```
